[L29] ## How to Integrate POS Systems Through API Requests

[L30] 

[L31] Most POS systems have an API library which OptiSigns can use to get the relevant data from the system programmatically. This API can return menu items, item price, availability, and more.

[L32] 

[L33] With OptiSync, we can link APIs to the OptiSigns portal and push the data to screens as a Digital Menu Board (DMB) or any other type of screen you'd like without the need of human intervention.

[L34] 

[L35] ![api optisigns integration diagram](https://support.optisigns.com/hc/article_attachments/31860108901523)

[L36] 

[L37] This article will focus on these POS specific wrinkles, and the process of mapping POS data to assets and pushing them to screens.

[L46] ### Get API Endpoint URL and Set Up API Request DataSource

[L47] 

[L48] We have a [comprehensive guide](https://support.optisigns.com/hc/en-us/articles/22875592994195-How-to-Integrate-API-and-Publish-API-Data-via-OptiSync) on how to set up your API gateway request. We recommend following this guide until your initial request is set up.

[L49] 

[L50] Bare minimum, you'll need an API endpoint URL and an API Authentication token.

[L51] 

[L52] ### Additional Information on API Authentication

[L53] 

[L54] For most token based authentication, setting up the authentication token with the key store is normally all that's required for an API request. But certain APIs (such as Toast) will require additional calls to get the authentication token for each request, this can be handled through pre-request processing. To see how to handle that, see our [article on Toast APIs](https://support.optisigns.com/hc/en-us/articles/31113088917907-How-to-use-Toast-API-data-with-OptiSigns).

[L56] ### Handling Multiple Stores or POS Locations

[L57] 

[L58] Once you've got your basic API Gateway request set up, there are a few additional steps you'll want to perform if you have multiple locations for your screens. These different locations may have different menus, or different specials for that day, or even different pricing depending on various factors.

[L59] 

[L60] POS systems normally require separate license for each location. Your POS system API may provide different store ID in the API endpoint or using different authentication token. For larger deployment with multiple stores, you can use substitution parameters to handle that with OptiSigns.¬Ý

[L61] 

[L62] There are two ways to handle multiple POS locations:

[L63] 

[L64] 1. Set up individual API requests for each of your POS locations, changing the value in the URL endpoint each time and mapping them to each of your screens individually. If you only have a few locations where your POS system is used, this will work just fine.

[L65] 2. *(Recommended)*Configuring each screen to send its storeID to the API call, allowing a single API request to provide data to multiple screens. For anything more than two or three screens, we recommend this method.

[L66] 

[L67] Here's how to handle option 2.

[L68] 

[L69] To get started, find the screen you wish to edit.

[L70] 

[L71] ![edit screen](https://support.optisigns.com/hc/article_attachments/31893086724755)

[L72] 

[L73] Click **Advanced** **‚Üí** **More** **‚Üí** **Device Additional Attributes.**

[L74] 

[L75] ![device additional attributes on edit screen](https://support.optisigns.com/hc/article_attachments/31893080684563)

[L76] 

[L77] Two fields will show up, **Key** and **Value**.  

[L78] ![device additional attributes key value](https://support.optisigns.com/hc/article_attachments/32043124363155)

[L79] 

[L80] * **Key** - A parameter that will be used during the API call to substitute for your store's value. This will replace part of your API URL endpoint.

[L81] * **Value** - Represents the unique code associated with the store or location you wish to pass through to your API.

[L82] 

[L83] In this example, we'll pretend the parameter you are changing is called "merchantID". The value inputted will need to be obtained on your end as it will be unique.

[L84] 

[L85] Now, go back to the API request config page. Substitute the merchantID in the API endpoint with the Key name you previously defined.

[L86] 

[L87] #### **clover url request**

[L88] 

[L89] When the API request is triggered on the device, it will take the value from the device and substitute it at runtime. For each screen, you'll want to perform these same steps, keeping the Key name the same while changing the Value. This will allow you to push different data to different screens off a single API Request.

[L91] ### How to Use Post-Request Processing to Convert API Data

[L92] 

[L93] When retrieving data from your POS system, it may not initially show up exactly the way you'd like, or you might want to add some functionality, such as the ability to display SOLD OUT for items out of stock.

[L94] 

[L95] For example, prices may display as whole numbers (i.e. 1299 instead of $12.99). That's where the "Post-request" tab comes in - this allows changes to be made to the data after it comes in. This will require some basic coding to use.

[L96] 

[L97] Take the example of the price display from earlier. How would we convert a number like 1299 to display as $12.99, and make that piece of code extensible to any similar display errors (e.g. 1899 instead of $18.99)?

[L98] 

[L99] ![](https://support.optisigns.com/hc/article_attachments/31893086743187)

[L100] 

[L101] For this common example, this piece of JavaScript code should solve your issue.

[L102] 

[L103] ```

[L104] let {data, headers, status} = os.context.get("response");  

[L105] temp\_data = data.elements  

[L106] for (let object of temp\_data) {  

[L107]  object.price = '$' + (object.price\*.01);  

[L108]  if (object.available == true)  

[L109]  {object.soldout=0;}  

[L110]  else {object.soldout=1;}  

[L111]  }  

[L112] return temp\_data

[L113] ```

[L114] 

[L115] This will fix the returned data, allowing it to display properly. It will also allow for creation of SOLD OUT and strike through for when items are out of stock.

[L116] 

[L117] ![](https://support.optisigns.com/hc/article_attachments/32060273039763)

[L127] ## How to Build Digital Menu Boards in Designer with OptiSync

[L128] 

[L129] In order to create a DMB with data from your POS system, the API Request will need to be registered as a DataSource. This allows data elements to be added to [OptiSigns' Designer app](https://support.optisigns.com/hc/en-us/articles/4404151402899-How-to-use-OptiSigns-Template-Designer-app-to-make-your-Digital-Signs-in-minutes), where it can be formatted and incorporated into any visual design you'd like to display.

[L130] 

[L131] The Designer app and templates can be used to do the formatting, and mapping the element to the data from the API data source. Any text box or image element can be mapped in Designer. When you map an image element, the URL of the image will be replaced at run time.

[L219] #### Creating Strike Throughs and Sold Out Warnings

[L220] 

[L221] In the above example, we show a Sold Out warning. However, we don't want to display that all the time - only when the item isn't available. With OptiSync, this can be accomplished thanks to the Post-request processing we did earlier. Our code created this **"soldout: 0"** data. This is tied to our **"available"** data:

[L222] 

[L223] ![](https://support.optisigns.com/hc/article_attachments/32077278913043)

[L224] 

[L225] When the "available" data reads "true," the "soldout" data reads 0. When your POS system detects items are unavailable, the "available" data will read "false". This will make the "soldout" data read 1.

[L226] 

[L227] We can use this knowledge to set up our Sold Out warnings and strike throughs to only appear when items are not available.

[L228] 

[L229] Going back to our Repeater Editor, we can click on a piece of text we want to strike through and hit **Settings**:

[L230] 

[L231] ![](https://support.optisigns.com/hc/article_attachments/32077293399315)

[L232] 

[L233] In the Settings tab, hit **Advanced Options**.

[L234] 

[L235] ![](https://support.optisigns.com/hc/article_attachments/32077801189779)

[L236] 

[L237] Under Advanced Options, hit **Property Mapping**.

[L238] 

[L239] ![](https://support.optisigns.com/hc/article_attachments/31968071408915)

[L240] 

[L241] Two values will show up: **Property** and **Value**.

[L242] 

[L243] ![](https://support.optisigns.com/hc/article_attachments/31968059040275)

[L244] 

[L245] Under Property, choose **Linethrough**.

[L246] 

[L247] ![](https://support.optisigns.com/hc/article_attachments/31968450832915)

[L248] 

[L249] Under Value, choose **.soldout.** Before the "." will be the name of your API Request.

[L250] 

[L251] **![](https://support.optisigns.com/hc/article_attachments/32077293403411)**

[L252] 

[L253] This sets the text to be crossed out when the "soldout" data reads 1.

[L254] 

[L255] We can repeat this with the Sold Out reading, except instead of Linethrough, choose **Visible**.

[L256] 

[L257] ![](https://support.optisigns.com/hc/article_attachments/31968463038227)

[L258] 

[L259] This will make the Sold Out text only appear when the "visible" data reads 1 - in other words, when your product is sold out.
