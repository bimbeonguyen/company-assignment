[L25] ## What API Integration Can Do

[L26] 

[L27] OptiSigns allows easy integration with user APIs. This allows data from a user's API to be shown on any of your digital signs. This allows dynamic updates for:

[L28] 

[L29] * **Digital Menu Boards** - Integrate with your POS systems and get the DMB updated automatically, and edit the format of your DMB using the designer app whenever needed.

[L30] * **Automated HR Update** - Use the API from your HR systems to display birthdays or work anniversaries on the screens automatically.

[L31] * **Warehouse Inventory Tracking** - Use your Warehouse's API to display inventory levels with automatic updates throughout your organization

[L32] * Any other information display use cases that you will need to consume API data and display the data on the screens.

[L33] 

[L34] This API integration can be achieved in OptiSigns with little to no coding, making it approachable for those without backgrounds in software engineering. In this guide, we will walk you through how to get it set up with the example below using an API from the Clover POS system.

[L51] The API gateway is a powerful tool that allows users to centrally manage the APIs, as well as configure and test the APIs.┬Ā

[L75] ### Step 1: Store Your API Authorization Token in the OptiSigns API Keystore

[L76] 

[L77] The first step is to take your API Authorization Token and store it in the OptiSigns API Keystore.

[L78] 

[L79] This step is technically optional: you can input your API Authorization token into an individual API request. However, the Keystore has a few major advantages:

[L80] 

[L81] * Allows your API Authorization token to be sent to multiple API requests, without the need to manually input it each time

[L82] * Provides superior security for your API Authorization token, making it much harder for outside actors to discover it

[L83] 

[L84] Therefore, we ***highly recommend*** this step.

[L85] 

[L86] To enter the Keystore, find the **Lock Icon** and click it.

[L87] 

[L88] ![firefox_YNrLQDITUl.png](https://support.optisigns.com/hc/article_attachments/32427362113683)

[L89] 

[L90] This will open the **API Keystore.**

[L91] 

[L92] ![](https://support.optisigns.com/hc/article_attachments/32427378817171)

[L93] 

[L94] Click┬Ā**Add Key**.

[L95] 

[L96] ![firefox_Af1bBuAYxy.png](https://support.optisigns.com/hc/article_attachments/32427378823187)

[L97] 

[L98] There are two fields here.

[L99] 

[L100] * **First Field -** Enter the name of the key here. We recommend something that will help you remember it. You'll be using this to set up an API request.

[L101] * **Second Field -** The actual unique passcode for your API communications.

[L102] 

[L103] Once you've input your API Authorization token, hit **Save**. When you want to run a request using this API Key, you'll use the term: **{{apiKeystore.<<key>>}}** where "<<key>>" is replaced by the name you inserted earlier. In this example, we'll name our request "clover".

[L104] 

[L105] ![](https://support.optisigns.com/hc/article_attachments/32427378828051)

[L106] 

[L107] Now, we're ready to set up your API request.

[L111] ### Step 2: Set Up and Test the API Request

[L112] 

[L113] Before moving forward in OptiSigns, we recommend [**testing your API**](https://www.postman.com/api-platform/api-testing/) connection using a free tool, such as [**Postman**](https://www.postman.com/). This provides several advantages, including:

[L114] 

[L115] * Checking your data formatting

[L116] * Ensuring the correct data is being provided

[L117] * Verifying your authentication

[L118] * Identifying integration problems or connection errors

[L119] 

[L120] The test parameters, endpoints, and authenticators can then be used in OptiSigns to set up your API request. Here's how to do that.

[L121] 

[L122] ![firefox_orSTybxxGU.png](https://support.optisigns.com/hc/article_attachments/32427362124563)

[L123] 

[L124] ┬Ā

[L125] 

[L126] Click the **Add Request** button, it will launch the window for you to configure and test the API request.

[L127] 

[L128] ![firefox_owER9ex9xQ.png](https://support.optisigns.com/hc/article_attachments/32427362128147)

[L129] 

[L130] * **Display Name -** This will be shown under the API gateway list, mainly to help users identify the API request.

[L131] * **Name -** This is used as a reference to the API request, it is a technical name that will be used later in the path to refer to the API request data.

[L132] * **URL -** This is the API endpoint, you can use the GET or POST method depending on the API request, for example, if you are using GraphQL, then all requests will be using POST.

[L133] * **Params -** Allows you to define the parameters in your API request. You may see parameters in your API endpoint URL, these will be preceded by a "?" mark. These can be used in pre and post processing request code to further automate your API request.

[L134] * **Headers -** The header(s) of the API request. You will want to input your API Authentication token here.

[L135] * **Pre-request -** An optional piece of code which prepares the context before the API request. For example, you may need to call another API to get a short-lived authentication token before calling the API, or you need to calculate some variables to be used in the API request.

[L136] * **Post-request -** An optional piece of code which modifies the data received from the request. For example, if the data you receive is not exactly how you want it displayed, you can write a bit of code to modify it according to your digital display needs.

[L137] * **Enable Webhook -** Generates a webhook link and an associated token. This webhook can be used to notify us of a change in the data, which will refresh the API request and update your screens automatically.

[L138] * **Enable this Request** - Set the status of the API request.

[L139] 

[L140] To test the request, we'll need to configure the header. This is where the Keystore comes in. In the second box, type **Bearer {{apiKeystore.<<key>>}}**, where *Bearer* is the type of token and *{{apiKeystore.<<key>>}}* pulls the token stored in the Keystore. In this example, we'll use the name "clover" as referenced above.

[L141] 

[L142] Once that's done, click **"Run Test"**. If the response code is 200, the API has returned data successfully. If there is any other code, there is an issue in the API Request.

[L143] 

[L144] ![](https://support.optisigns.com/hc/article_attachments/22917836593171)

[L146] #### How to Use Parameters

[L147] 

[L148] Parameters are present in URLs of all types. There are two elements to a parameter:

[L149] 

[L150] * They have to follow a '?' mark in the URL;

[L151] * They have a value associated with them

[L152] 

[L153] The **Params** tab lets you specify the Parameters whose value you would like to change.

[L154] 

[L155] Usually, the **Params** tab is used in conjunction with the Pre-request and Post-request tabs to create automatically updating values.

[L156] 

[L157] #### How to Use Pre- and Post-Request Processing

[L158] 

[L159] To use pre- and post-request processing, some amount of Javascript knowledge is needed.

[L160] 

[L161] |  |

[L162] | --- |

[L163] | What is the Difference Between the Two? |

[L164] | **Pre-request:** This is a code snippet normally set the context ***before*** the API request. This can be retrieving an authentication token from another API, or to change parameters to allow for more automation. |

[L165] | **Post-request:** A piece of code to apply to the data ***received from*** the API request. This code can be used to modify the received data to change how it is displayed on your screens. |

[L166] 

[L167] ┬Ā

[L168] 

[L169] The **Pre-request** tab is where you'll input code for pre-request processing.

[L170] 

[L171] **Example:** For the systems that requires a dynamically generated authentication token like Toast, this can be used to call another API to retrieve the authentication token and set it to the context of the API.

[L173] For more on this example, see this article on [how to connect your Toast API](https://support.optisigns.com/hc/en-us/articles/31113088917907-How-to-use-Toast-API-data-with-OptiSigns).

[L177] The **Post-request** tab is where you'll input code for post-request processing.

[L197] ### Step 3 (Optional): Use Substitution Parameters from Device Attributes.

[L198] 

[L199] Many point-of-sale (POS) systems are licensed by store/location. It's possible to configure a single API Request with OptiSync and have it work with different locations using **substitution parameters**. Inputting these allows your screen to communicate its store or location identification information. This means a single API request can communicate different data to different stores or locations, rather than needing to make individual API requests per screen.

[L200] 

[L201] To get started, find the screen you wish to edit.

[L202] 

[L203] ![](https://support.optisigns.com/hc/article_attachments/32427362139411)

[L204] 

[L205] Click **Advanced** **ŌåÆ** **More** **ŌåÆ** **Device Additional Attributes.**

[L206] 

[L207] ![](https://support.optisigns.com/hc/article_attachments/32427378848019)

[L208] 

[L209] Two fields will show up, **Key** and **Value**.  

[L210] ![firefox_KkCBvxsPKU.png](https://support.optisigns.com/hc/article_attachments/32427362146707)

[L211] 

[L212] * **Key** - A parameter that will be used during the API call to substitute for your store's value. This will replace part of your API URL endpoint.

[L213] * **Value** - Represents the unique code associated with the store or location you wish to pass through to your API.

[L214] 

[L215] In this example, we are maintaining the Clover merchantID here. The value inputted will need to be obtained on your end as it will be unique.

[L216] 

[L217] Now, go back to the API request config page. Substitute the merchantID in the API endpoint with the Key name you previously defined.

[L218] 

[L219] #### 

[L220] 

[L221] When the API call is triggered on the device, it will take the value from the device and substitute it at runtime. For each screen, you'll want to perform these same steps, keeping the Key name the same while changing the Value. This will allow you to push different data to different screens.

[L225] ## How to Map API Data in Designer

[L226] 

[L227] In order to push data from your API to a screen, it will need to be registered as a DataSource. This allows data elements to be added to OptiSigns' Designer app, where it can be formatted and incorporated into any visual design you'd like to display.

[L228] 

[L229] The Designer app and templates can be used to do the formatting, and the API data source will handle the data mapping. Any text box or image element can be mapped in Designer. When you map an image element, the URL of the image will be replaced at run time.

[L230]
