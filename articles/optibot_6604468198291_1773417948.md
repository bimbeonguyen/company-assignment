[L1] # How to Use the CAP Alert App

[L2] 

[L3] ### In this article, we will show you how to set up and test the CAP Alert app in OptiSigns.

[L10] 

[L11] Some Emergency Alert Systems or Emergency Mass Notification Systems (like Everbridge, RAVE. and Alertus) can push CAP (Common Alerting Protocol) and Integrated Public Alert and Warning System (IPAWS) messages to the targets including digital signage when there is an emergency.¬†You can integrate with these systems using the CAP Alert app with OptiSigns.

[L12] 

[L13] Using OptiSigns' CAP Alert app, you can generate a webhook and integrate it with the Emergency Alert System. When there is an emergency, the emergency alert system will call the webhook to send the CAP/IPAWS message and trigger the CAP alert app. The CAP/IPAWS alert will take over the target screens and display the emergency message. The screen will resume and play the original content when the emergency is over.¬†

[L28] ## How to Set Up a CAP Alert App

[L29] 

[L30] Go to the OptiSigns portal. Go to **Assets ‚Üí Add Asset ‚Üí Apps.**

[L31] 

[L32] ![](https://support.optisigns.com/hc/article_attachments/48757530689299)

[L33] 

[L34] Select **CAP Alert:**

[L35] 

[L36] ![](https://support.optisigns.com/hc/article_attachments/48757530691731)

[L37] 

[L38] Now you can set up your Looker Studio app:

[L39] 

[L40] ![](https://support.optisigns.com/hc/article_attachments/48757530694419)

[L41] 

[L42] ¬†

[L43] 

[L44] * **Name** - Name of your assets, this will not be displayed on the screens.

[L45] * **Content Type** - Choose between **Post to Webhook** or **XML**.

[L46] * **Enable Authentication** - When checked, will add a **Username** and **Password** section to check Authentication.

[L47] 

[L48] ![](https://support.optisigns.com/hc/article_attachments/48757530696723)

[L49] * **Webhook / XML** - Depending on whether you've selected "Post to Webhook" or "XML" above, here is where the webhook or XML script will be placed.

[L50] * **Target** - Choose whether this alert will target a specific screen, or tag.

[L51] * **Screens/Tags** - Select which screens or group of screens (tags) you want to target for this emergency. (i.e. Fire in building/location 1)

[L52] * **Status** - Swap between Active or Inactive for this alert.

[L53] * **Emergency Duration** - How long the CAP Alert will take over the screen. Measured in seconds.

[L54] * **Content-Type** - Select "Post to Webhook" if you would like to post the CAP/IPAWS message to your signage. The app also supports RSS feed.

[L55] * **Webhook** - The app will generate a webhook URL after it is saved. This is what you should share with the emergency alert system.

[L56] * **Display Type** - Currently the app will take over the full screen when there is an emergency

[L57] * **State**- Set the app to active or inactive.

[L58] * **Emergency Duration**- How long the emergency message will take over the screen. The value can be overwritten by the webhook call.

[L80] ### Advanced Settings

[L81] 

[L82] Click **Advanced** to expand the field and provide a slate of additional options:

[L83] 

[L84] ![](https://support.optisigns.com/hc/article_attachments/48757530712979)

[L85] 

[L86] * **Enable Lifecycle Handling** - When checked, it allows handling of CAP message lifecycle events including Update, Cancel, and All Clear messages. It does this by checking the <identifier> field. **This option is only applied when "Post to Webhook" is set as the Content Type**. It also enables the below options:

[L87] 

[L88] ![](https://support.optisigns.com/hc/article_attachments/48786708018707)* **Handle All Clear** - When enabled, if the alert system sends an "All Clear" message, a brief "all clear" notice will display on all screens for 60 seconds by default. Once this time has passed, content will resume as normal.

[L89] 

[L90] |  |

[L91] | --- |

[L92] | **MORE ABOUT ALL CLEAR** |

[L93] | OptiSigns handles All Clear messages in this way:1. First, it finds the <responseType> in the message:

[L94] `<responseType>AllClear</responseType>`

[L95] 2. Then, it checks the <expires>:

[L96] `<expires>2027-01-31T11:23:46-05:00</expires>`

[L97] 3. If AllClear is noted in the response type, it will display the message **until the designated expires time**. If no time is sent, it will default to 60 seconds.

[L98]  |

[L99] 

[L100] * **Handle Update -** When enabled, the app will automatically refresh screens with additional information on the emergency alerts as updates become available.

[L101] * **Handle Cancel** - When enabled, the app looks for a "Cancel" message referencing a previous alert. If detected, the emergency alert is immediately removed from your screens and normal content resumes.

[L102] * **Item Duration** - How long each individual item in the CAP Alert feed will display.

[L103] * **Title Tag** - Message title from the CAP/IPAWS message/RSS feed. Default is <headline> - you can change if your feed is different

[L104] * **Description Tag** - Message content from the CAP/IPAWS message/RSS feed, default is <description> - you can change if your feed is different

[L105] * **Location (Screen Tags)** - If you can match the screen tags with your location passed from the CAP/IPAWS message/RSS feed - you can use it to control the selection of the target screens. By default, it maps to the "areaDesc" attribute from the CAP alert.

[L106] 

[L107] |  |

[L108] | --- |

[L109] | **IMPORTANT** |

[L110] | A common issue we find is the screen displays a "No Content Available" message after users push out the CAP alert using "aeraDesc" an attribute. The solution: if you are not intending to use screen tags to map to location, try changing the Location value from "areaDesc" to any other value, like "areaDesc2". |

[L111] 

[L112] ![](https://support.optisigns.com/hc/article_attachments/48757514534803)

[L113] 

[L114] * **Severity, Urgency, Certainty** - Standard attributes of CAP / IPAWS messages, these options allow you to control the filter of the messages. By default, the app will be triggered on all values. However, you can set these similar to tags - this will filter out anything that does not include these tags.

[L115] * **Filter content containing** - Allows content to be filtered based on specific words in the title or description. I.E: "fire", so if any title or description contains the word "fire" (non-case insensitive), the app will trigger the screen takeover.

[L116] * **Exclude title containing** - Filter only applying to the title. You can hide all the old feeds by filtering with specific words in the title. I.E: ‚ÄúAll Clear‚Äù, so after the emergency is gone, all the feeds before this title will be hidden, then the screen will revert to the original content or just display the new content after that.

[L117] * **Category, Code, Event Code** - Additional filters allowing you to include or exclude certain event categories, codes, or event codes.

[L118] 

[L119] When finished, click **Save**.

[L123] ## How to Test Your CAP Alert

[L124] 

[L125] After Saving, the app will generate a Webhook URL automatically. When this Webhook is called, it will perform the functions designated in your CAP Alert app. If you share the Webhook URL with your Emergency Alert System, it will be able to trigger this Webhook during an emergency and take over the screens.  

[L126] ¬†

[L127] To test your CAP alert integration, we recommend using [Postman](https://www.postman.com/) to post a CAP/IPAWS message to the Webhook.

[L128] ¬†

[L129] First, download or log in to Postman. Once there, navigate to **Home ‚Üí Send an API request ‚Üí New Request:**

[L130] 

[L131] **![postman with arrows pointing toward Home tab and New Request button](https://support.optisigns.com/hc/article_attachments/33695686427411)**

[L132] ¬†

[L133] Once here, change the request type to **POST:**

[L134] 

[L135] ![postman demonstration of how to select POST option](https://support.optisigns.com/hc/article_attachments/33695686435475)

[L136] Input the Webhook URL for your CAP Alert next.

[L137] ¬†

[L138] Now, navigate to the **Body** tab. Here, you can send data two ways: **URL encoded** or as a **raw XML file**. Here we will assume you are sending your data as raw XML, as this is the most common format we see from Emergency Alert Services.

[L139] ¬†

[L140] CAP/IPAWS Alerts are generally delivered via an external Emergency Alert System, which are saved as **.xml** or **.txt** files. These are broadcast via the [Common Alerting Protocol Version 1.2](https://docs.oasis-open.org/emergency/cap/v1.2/CAP-v1.2-os.html). To push an XML file, click the **raw ‚Üí Dropdown ‚Üí** select **XML**.

[L141] **![postman step by step on selecting XML format](https://support.optisigns.com/hc/article_attachments/33695686448403)**

[L142] ¬†

[L143] Then, copy and paste the following piece of test code into the field:

[L144] ¬†

[L145] 

[L146] ```

[L154] ¬† ¬† <source>Alertus eEAS Server</source>

[L155] ¬† ¬† <scope>Public</scope>

[L156] ¬† ¬† <addresses>ALL</addresses>
